# TODO — Express.js Backend: Optimize Placement Feature


## Overview
New endpoint `POST /v1/ai/optimize-component` that receives a single component's snapshot and returns 3–5 alternative plant arrangements. Arrangements are generated by a **deterministic greedy heuristic + perturbation algorithm** and scored with the plant placement scoring model from `src/data/plant-placement-scoring.md` (synced to the backend).

**Distinct from `POST /v1/ai/optimize-layout`** (garden-wide, LLM-driven, move-based). This endpoint is:
- Component-scoped (one component, not a garden)
- Deterministic (no LLM in the hot path for v1)
- Returns complete absolute position sets, not move deltas
- Returns 3–5 alternatives for user comparison, not one proposal

See `src/data/optimize-placement-architecture.md` for the full ADR.

## Phase P — Scoring Engine

- [ ] **P1** — `src/scoring/PlantScoringEngine.ts`
  Pure, side-effect-free TypeScript module. Constants:
  ```typescript
  const PLANT_INNER_RADIUS_CM = 5;
  const INTERACTION_FACTOR = 2;
  ```

  **`scoreSpacing(plants, catalog): number` — 0–100**
  Count non-patch pairs. For each pair where distance < max(r1, r2): spacing violation.
  Score = 100 * (1 - violations / totalPairs). If totalPairs = 0, return 100.

  **`scoreCompanions(plants, catalog): number` — 0–100**
  Benefit point values (from plant-placement-scoring.md):
  - fixesNitrogen: 20 pts
  - detersPests: 15 pts
  - attractsPollinators: 15 pts
  - growthBoost: 10 pts
  - improvesFlavor: 5 pts
  For each non-patch pair within INTERACTION_FACTOR x max(r1, r2): sum benefit points.
  Normalise: rawScore / maxPossibleScore x 100. Cap at 100. Return 50 if no companion pairs possible.

  **`scoreCombatives(plants, catalog): number` — 0–100 (higher = fewer conflicts)**
  Harm penalty values:
  - diseaseRisk: 30 pts
  - inhibitsGrowth: 20 pts
  - depletesNutrients: 15 pts
  - attractsPests: 15 pts
  For each non-patch pair within interaction radius: sum harm penalties.
  Score = max(0, 100 - totalPenalty).

  **`scoreSun(plants, catalog, componentSunDirection): number` — 0–100**
  Direction to light level: S/SE/SW = full sun, E/W = partial, N/NE/NW = shade.
  Per-plant score table:
  - full sun plant in full zone: 100 | partial zone: 50 | shade zone: 0
  - partial plant in full zone: 80 | partial zone: 100 | shade zone: 100
  - shade plant in full zone: 20 | partial zone: 60 | shade zone: 100
  Final = mean of all individual scores (including patch plants).

  **`scoreTotal(breakdown, objective): number`**
  Weights per objective:
  - balanced: companion 30% + spacing 30% + sun 25% + combative 15%
  - maximize-companions: companion 50% + spacing 25% + sun 15% + combative 10%
  - minimize-harm: combative 50% + spacing 25% + companion 15% + sun 10%

- [ ] **P2** — Unit tests for `PlantScoringEngine` (`tests/scoring/plantScoringEngine.test.ts`):
  - Known companion pair (tomaat + basilicum) within 2 x max(spacing) — scoreCompanions > 50
  - Known combative pair (tomaat + broccoli) within range — scoreCombatives < 100
  - All plants uniformly spaced beyond spacingRadius — scoreSpacing = 100
  - All plants on S-facing component with sun 'full' — scoreSun = 100
  - All plants on N-facing component with sun 'full' — scoreSun = 0
  - 0 plants — all scores return neutral values (100 spacing, 50 companions, 100 combatives)

## Phase Q — Placement Optimizer

- [ ] **Q1** — `src/scoring/PlacementOptimizer.ts`
  Generates `numberOfAlternatives` (3–5) distinct candidate layouts.

  **Inner bounds calculation:**
  - rect: (widthInCm - 2*border) x (lengthInCm - 2*border)
  - circle: diameterInCm - 2*border as radius r = diameter/2; disc validity: (x-r)^2 + (y-r)^2 <= (r - PLANT_INNER_RADIUS_CM)^2
  - tower layer: apply 0.85^layerIndex reduction to inner dims

  **Validity checks per position:**
  - rect: PLANT_INNER_RADIUS_CM <= x <= innerWidth - PLANT_INNER_RADIUS_CM (and same for y)
  - circle: disc equation above
  - hard collision: distance between two non-patch plants >= 10 cm

  **Algorithm:**
  1. Seed layout (greedy, companion-aware):
     Sort singletons by descending spacingRadiusCm. For each, evaluate 5 cm grid of candidate positions.
     Cell score = companion_bonus(neighbours) - hostile_penalty(neighbours) - spacing_comfort_penalty - boundary_penalty.
     Place at argmax. Reject cells with hard collision or out-of-bounds.
     Patch plants keep their current positions.

  2. Perturbation (~30 candidates, 4 strategies round-robin):
     a) Pair swap: swap (x,y) of 2 random non-patch singletons
     b) Local jitter: nudge 1-3 plants by U(-15, +15) cm; clamp, push-off-wall
     c) Companion pull: move one singleton halfway toward a companion in current layout
     d) Hostile push: move one singleton 2x spacing away from nearest hostile; clamp
     Reject candidates failing validity. Resample <= 3 times per strategy.

  3. Score all candidates with PlantScoringEngine.scoreTotal(objective).

  4. Diversity-aware top-N:
     layoutDistance(A, B) = average per-plant Euclidean displacement.
     Greedy farthest-first: alt-1 = highest scorer; alt-N+1 = highest scorer with layoutDistance > 15 cm
     from all selected (relax to 10 cm if needed).

  5. Determinism: seed PRNG from hash(componentId + sorted(plantInstanceIds) + objective).

- [ ] **Q2** — Unit tests for `PlacementOptimizer` (`tests/scoring/placementOptimizer.test.ts`):
  - Output has exactly `numberOfAlternatives` entries (or fewer with diagnostic warning)
  - All positions within inner bounds
  - No hard collision for non-patch pairs (distance >= 10 cm)
  - All plant instance IDs present in every alternative (cardinality preserved)
  - Alternatives are distinct (avg displacement > 10 cm between any two)
  - Tower plants keep layerIndex
  - Patch plants not repositioned
  - Same inputs produce same output (deterministic)

## Phase R — Endpoint

- [ ] **R1** — `POST /v1/ai/optimize-component` — hardcoded stub response, no validation yet

- [ ] **R2** — Add `validateBody(optimizeComponentRequestSchema)` middleware

- [ ] **R3** — `src/controllers/optimizeComponentController.ts`:
  1. Extract `componentSnapshot` of the target component (find by `componentId` in `snapshot.components`)
  2. Return 400 if `componentId` not found in snapshot
  3. Call `PlacementOptimizer.generate(componentSnapshot, plantCatalog, objective, numberOfAlternatives)`
  4. Enrich each alternative with `label` and `summary` (see R4)
  5. Return `OptimizeComponentResponse`

- [ ] **R4** — `src/prompts/optimizeComponentSummary.ts` — Dutch **template** summary (no LLM in v1):
  Base template: "Optie {n}: score {total}/100 — {numCompanionPairs} gezelschapparen actief, {numViolations} afstandsoverschrijdingen, {numHostilePairs} conflicten in bereik."
  Extra prefix when total >= 70: "Uitstekende opstelling."
  Extra suffix when companion >= 80: "Maximale samenwerking tussen planten."
  Warning suffix when combative <= 20: "Let op: conflicterende planten staan dicht bij elkaar."

- [ ] **R5** — Post-validation: after optimizer output, re-run all validity checks server-side. Discard failing alternatives. Append diagnostics.warnings entry per discard. If zero alternatives survive: return 400 with code OPTIMIZE_NO_VALID_ALTERNATIVES.

- [ ] **R6** — Mount the new router; update `GET /v1/ai/health` to include `"optimizeComponent": true` in the `features` object.

- [ ] **R7** _(FUTURE LLM SWAP)_ — Replace template `summary` with LLM-generated Dutch summary via OllamaClient/LlamaCppClient when LLM_MODE != mock. Graceful degradation to template on LLM timeout. Summary prompt in `src/prompts/optimizeComponentSummaryPrompt.ts`.
